The computing industry is weird about history. People fail to understand it in two opposite ways:

A lot of things just fail to pay attention to the fact that other people have seen and solved the same problem before. You get people not just reinventing the wheel, but ignoring the huge design space of wheels that’s been explored and deciding that cubes are ideal shapes for wheels and concrete the ideal building material. See pretty much every recent GUI for hundreds of examples.

But at the other extreme you get a weird worship of the past. Smart people in the past did something this way, therefore we must do things that way! The fact that their target audience was PhD students and their constraints included a processor that was barely faster than a pen and paper and had 128 KiB of RAM is irrelevant. Early UNIX did glob expansion in the shell not because that’s more sensible than providing a glob and option parsing API in the standard library, but because they didn’t have enough disk space or RAM to duplicate code and they didn’t have shared libraries. If you have shared libraries, the right choice is very different. Similarly, ‘everything is a text stream’ is a good idea when you have a computer that you connect to via a serial terminal because everything that the user produces or consumes is a text stream. When you have a system that has graphical displays, speakers, cameras, microphones, and network connections, it is much less sensible.

Solid colors clearly delineate controls

More importantly, the visual metaphor of a button is universal. Anything you can press (click on) has a bevel around the edge that makes it appear raised. This includes the menus. Every menu item is a button, so you don’t need to learn anything about menus to know that you can click on them. Menus that expand to submenus have an arrow, which you do need to learn, but it’s discoverable as soon as you click on one (which you know you can because of the bevel). This is even carried to window title bars, which are raised in a similar way to indicate that you can drag them.

It’s not visible in this screenshot, but anything that you place data into is sunk, to represent a hole in the UI that you can fill in. Text fields, for example, have the opposite bevel to a button. Colour wells combine these because they are a drop target for colours (you can drop a colour in them to set the colour) but they are also a button that you can press, so they look like a caldera: a raised area with a coloured hole in the middle.

Newer GUIs seem to have forgotten that the purpose of visual design is to convey information in a way that reduces cognitive load. You can look at a NeXT window and see exactly what you can click on, what you can enter data into, and what is purely informative, without thinking. The design is so consistent that you don’t even realise that you’re making the connection.

With OS X 10.0, Apple maintained this (mostly: there were rough corners where Cocoa / OpenStep visual metaphors clashed with Carbon / Classic MacOS ones). Clickable things were both raised and also coloured like boiled sweets. Informative things were on grey backgrounds, things you interacted with were white. It wasn’t quite as good in a few places, but it mostly worked. Modern macOS (even before Liquid Glass, which is a total disaster) has gradually lost a lot of this consistency.

When we designed the Nesedah theme as an updated NeXT look, we used gradients to indicate anything that you would interact with. It worked pretty well. I tried showing a few people mockups and they were able to identify all of the UI components without my explaining them. If I had been doing it for a corporation, I’d have done proper user studies to make sure. It’s quite clear MS and Apple are not doing that anymore.

the Root server system consists of 13 logical root name servers, each consisting of globally distributed and redundant network of physical instances of that root name server. they use a technique called "anycast routing" where multiple network operators will route the same IP addresses; and requests to a given root name sever are generally handled by one of the closest physical instances of that root server.

K-root is operated by RIPE NCC and it returns such diverse results because each of its physical instances may run run one of BIND, NSD, or Knot DNS, and each query i make is probably answered by a different physical server running different software.

in fact, i know a different K-root server answered each of my queries because some of my queries resolve in about 15 milliseconds, and some resolve on the order of 140 milliseconds. i suspect that the shortest 15ms response time is K-root in Hamburg operated by IPHH Internet Port Hamburg GmbH, because it is the closest one to my location in the Västra Götalandsregionen of Sweden. and, i suspect that the queries on the order of 140ms may be going through the North Sea Link undersea cable and hitting one of the K-roots in London operated by RIPE NCC. your results with K-root are going to differ, because it contains over 142 sites/locations; many of which in turn contain multiple instances of K-root. you can see a map of all K-root nodes on their website.

the software diversity of K-root can be contrasted with D-root and H-root, which according to Wikipedia both run just NSD, and if i query them i can sure see that they return identical results in alphabetical order, then IPv6 addresses in alphabetical order, then IPv4 addresses in alphabetical order. therefore, it seems like an accurate claim that they do in fact run identical server software.

likewise, the geographic diversity of K-root can be contrasted with the sparseness of B-root, having only 6 physical instances, causing my queries to B-root to resolve very consistently in about 18ms. that latency leads me to believe that i'm probably hitting the server in Frankfurt am Main every time, because it checks out with K-root responding slightly faster, because its location in Hamburg is physically closer to me, beating B-root by about 3 milliseconds on a good day.

G-root is just as geographically sparse as B-root, also having just a single instance in my vicinity. but G-root's closest site is in Stuttgart, so it takes an additional 4 milliseconds or so to answer my queries, compared to B-root.

taking this to its extreme, we can single out the japanese operator of M-root. the official root server metrics website tells me that M-root has a single European point of presence in Paris, but given that every single one of my queries take over 250 milliseconds to be answered, i can reasonably conclude that M-root's Paris instance is unavailable right now. but the anycast redundancy is working as intended, because my queries are still being answered even though the Paris instance is likely offline at this time.

but i don't need to hypothesize; i can actually verify that the Paris instance of M-root is not answering my queries by querying something like dig +norec @m.root-servers.net id.server chaos txt. this tells me that i'm always talking to M-NRT-JPNAP-1. the JPNAP in that name refers to a major internet exchange in Japan. the M-root homepage says that JPNAP is responsible for M-root instances in Tokyo and Osaka, but i can be reasonably sure that M-NRT-JPNAP-1 (the only one that i ever got answers from) is actually located in Tokyo because today on RIPE Atlas probe 6425 reports that M-NRT-JPNAP-1 responds in less than two milliseconds (note: i can permalink a timestamp and filter, but not a specific probe. type "6425" into the searchbar). at 2025-08-28 06:30:04 UTC, that root server instance even responded to the probe in under a millisecond, giving a high degree of confidence that this one is located in Tokyo. and that explains why, when i query M-root from Sweden, it takes so damn long to answer. it's because every query goes to Tokyo and back!

on the opposite end of the latency spectrum for me are E-root, F-root, and I-root. these are the only 3 root servers with a physical presence in Göteborg, and my queries to these root servers are answered in less than 10 milliseconds, without fail.

F-root is primarily operated by the Internet Systems Consortium, but since 2017 has cooperated with Cloudflare, allowing their extensive global network to also announce routes for F-root. Because of Cloudflare's extensive network providing F-root instances, it is the single largest root server network as of today, and chances are you have an F-root near you! Yes, you!
